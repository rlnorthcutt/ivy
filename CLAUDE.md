# CLAUDE.md

## Project Overview

Ivy.css is a classless CSS baseline library that provides elegant, opinionated defaults for semantic HTML elements, structured as a two-layer system (core + extra) with an optional utility companion (Lattice).

## Architecture

### Two-Layer Design
- **`ivy.css`** (core): Classless defaults for all native HTML elements. Uses CSS custom properties for a comprehensive design token system with light/dark theme support and fluid responsive typography. No classes required — semantic HTML just works.
- **`ivy.extra.css`** (extra): Optional enhancement layer. Adds component-level styling (buttons, tooltips, dialogs, badges, cards, callouts, progress bars, spinners) via `data-*` attribute APIs rather than classes.

### API Convention
Optional patterns are activated via **data attributes**, not class names. This preserves the classless philosophy while enabling opt-in components.

### Build Output
- `dist/` — standalone minified files for package distribution
- `docs/` — minified files used directly by the GitHub Pages documentation site

The CI workflow produces:
- `ivy.min.css` — core only
- `ivy.extra.min.css` — extra only
- `ivy.full.min.css` — core + extra combined bundle


## Development Conventions

- **No build step required locally** to work on source files — edit `ivy.css` or `ivy.extra.css` directly.
- Design tokens are CSS custom properties defined on `:root` and updated via `@media (prefers-color-scheme: dark)` or a `[data-theme]` attribute for manual toggling.
- Components in `ivy.extra.css` use `data-*` attributes as selectors (e.g., `[data-button]`, `[data-card]`), never class names.
- The extra layer mirrors the core token system and extends it with additional surface and shadow tokens — keep token naming consistent between the two files.
- Documentation pages (`docs/*.html`) serve as both visual demos and the live GitHub Pages site. They must be valid, semantic HTML to demonstrate the library properly.

## Common Tasks

### Minify CSS locally
Requires `clean-css-cli`:
```bash
npm install -g clean-css-cli

# Minify individually
cleancss -o dist/ivy.min.css ivy.css
cleancss -o dist/ivy.extra.min.css ivy.extra.css

# Combined bundle
cleancss -o dist/ivy.full.min.css ivy.css ivy.extra.css

# Copy bundle to docs for GH Pages
cp dist/ivy.full.min.css docs/ivy.full.min.css
```

### Serve docs locally
```bash
bash docs/devserver.sh
```

### Deploy
Push to `main` — the `minify-css.yml` workflow automatically minifies, commits output to `dist/` and `docs/`, and deploys to GitHub Pages.

## What to Avoid

- **Do not add class-based selectors to `ivy.css`** — it must remain classless. All element styling targets tag names or native pseudo-classes.
- **Do not use class selectors in `ivy.extra.css`** — use `data-*` attribute selectors for all opt-in components.
- **Do not manually edit files in `dist/` or the minified files in `docs/`** — these are build artifacts regenerated by CI on every push to `main`.
- **Do not add JavaScript to the core or extra CSS files** — the dark-mode toggle script is intentionally separate.
- **Do not use `defer` on `type="module"` scripts** — ES modules are deferred by default; the attribute is redundant.
- **Avoid bare relative paths** (e.g., `ivy-core.html`) in navigation links without considering base path changes if the docs move off the root.
- **Do not add write permissions to the workflow for PR triggers** — the workflow uses `GITHUB_TOKEN` with write access; keep triggers restricted to trusted branches (`main`) only.
